### §4.2 Δ-18 orchestrator-pattern(NEW)— 草签

**source**: hermes `orchestrator/{loop.py, agents.py, acceptance.py, charter.py, gates.py, checkpoints.py, state.py}` + `docs/proposals/{orchestration-protocol-draft,acceptance-agent-draft,mission-charter-template-draft,aidazi-workflow-governance-variant}.md`.

#### T0 universal(framework constants)

- **Mission Charter** = YAML/JSON pre-authorization artifact。schema:
  ```yaml
  mission: {id, goal}
  autonomy: {level, approved_scope: {subsprint_sequence, layers_allowed, modules_in_scope, explicitly_out_of_scope}, auto_pass_rules: {clean_pass_auto_advance, auto_fix_iteration: {enabled, max_rounds, only_if_findings_severity_at_most}, adaptive_insert: {enabled, max_inserted_subsprints}, acceptance: {enabled, mode, run_at, judge_model, calibration}}}
  budget: {max_api_usd, max_fix_rounds_total, ...}
  tooling: {dev: {agent_kind, model, sandbox}, review, eval, acceptance}
  ```
- **MANDATORY_CHECKPOINTS**(charter 只能 ADD 不能 REMOVE):
  1. mission_start
  2. research_proposal_selection
  3. bad_case_manual_review
  4. new_tier0_candidate
  5. forbidden_list_redline
  6. scope_deviation
  7. close_taxonomy_C_or_D
  8. gate_hard_fail
- **File-based checkpoint inbox**:`docs/checkpoints/<timestamp>__<reason>__<subsprint>.md`,human 写 `decision: approve|revise|stop`。
- **Scope envelope check**(deterministic,无 LLM):
  - subsprint sequence membership
  - layer set membership
  - explicitly_out_of_scope module hits
  - unknown module hits
- **F5 evidence pattern**:orchestrator(非 agent)跑 real-LLM eval harness 产出 evidence,acceptance judge **read-only** 读 evidence。dev sandbox 始终封闭。
- **§3.5 Acceptance calibration gate**:judge 在 (trace, expected_verdict) labeled set 上跑;`agreement_rate ≥ 0.9 AND flip_rate ≤ 0.1` → 标定 `calibrated`;否则 auto_iterate mode 自动降级为 `surface_approve`。

#### T1 — Type A driver implementation(hermes-style)

State machine:
```
idle
  ↓ (current_subsprint set)
dev_pending → gate_pending → review_pending → close_pending
                                                  ↓
                              route_close: advance | fix | checkpoint
                                                  ↓
              advance ← (next subsprint)  fix ← (back to dev_pending; bump fix_round)
                  ↓
            (last subsprint) → milestone_close → acceptance_pending
                                                       ↓
                              route_acceptance: pass | fix_required → plan_and_arm_fix → dev_pending
                                                       │
                                                       └→ needs_human → surface_approve checkpoint
```

Spawn functions(JSON-schema'd):
- `spawn_dev` — Codex / equivalent,workspace-write sandbox,无网络,无 `git push`
- `spawn_deliver_close` — LLM with verdict schema {A/B/C/D, worst_severity, next_subsprint}
- `spawn_deliver_plan_fix` — LLM with plan schema {subsprint_id, layers, modules, objective_md, dev_prompt_md}
- `run_review` — LLM read-only by **mechanical tool whitelist**(非 plan mode,plan mode 会把 verdict 藏掉)
- `run_acceptance` — LLM read-only by tool whitelist;Acceptance judge contract(详 §5 内 acceptance prompt 模板)
- `spawn_research` — 可选;hermes 当前 unused(human-paste pre-loop)

Gates(deterministic,无 LLM):
- `run_tests` / `run_eval`(F5)/ `validate_stanza` §7 / `check_handoff` §1-§11 + §12 empty / `check_trace` / `check_commit_discipline`
- skipped required gate = NOT passed

#### T1' — Type B driver(SOP variant,占位)

Type B 项目(hermes-style SOP-row workflow)需要 charter `approved_scope` 中加 `workflow_definition` layer;state machine 大致一致,但 review 关注 SOP step verification 而非 anti-hardcode;细节 placeholder 待 hermes 后续完整 lifecycle 验证。

#### Δ-18 anti-patterns(必写入 framework forbidden 增量)

- 直接编辑 MANDATORY_CHECKPOINTS 列表移除项
- run_acceptance 未 calibrated 就在 auto_iterate mode 信任 verdict
- 跳过 scope_envelope_check
- dev sandbox 读 shadow eval set
- LLM judge 从代码 inspection 而非 execution evidence 出 verdict
- charter 用一份默认 acceptance.mode = "auto_iterate" 同时 calibration = uncalibrated

## §7 — Open questions for Checkpoint 1(用户决定)

P3 启动前需要你裁决的 6 个设计点:

### Q1. closure_contract 的归宿

`closure_contract` 我提议放在 **Research brief 主体里**(`docs/research-briefs/<id>.md` 必填字段),Acceptance 后续读这份 contract 校验交付。备选:放在 milestone_objective 里(deliver 拥有)。**建议 Research 持有**:让 brief 既是 intake spec 又是 outcome contract,Acceptance 与 Research 同 level 才说得通。

### Q2. Acceptance 在 sub-sprint 级别 vs 仅 milestone 级别

hermes 当前 acceptance 是 milestone 级。我倾向 v4:
- 默认 **milestone close + release cut** 触发
- Sub-sprint close **可选触发**(Type C demo 强制,Type A/B 看 charter)
- 不要每 sub-sprint 都跑 — Acceptance 成本高(read-only LLM + evidence gathering + calibration check)

### Q3. Customer 与 Acceptance 的交互回路

Customer 收到 Acceptance report 后,若 push-back,routes 是:
- (a) push-back → Acceptance 重跑 cited criterion → 新 report → Customer 再决
- (b) push-back → Research 重新审视 closure_contract 是否本就不对(契约层 bug)
- (c) push-back → Deliver 直接接 gap

我建议三路并存,Acceptance report 显式建议 route。

### Q4. Δ-18 T1' Type B 现在抽到什么程度

hermes 是 A+B hybrid,SOP 部分尚未走完完整 lifecycle。**建议**:Δ-18 T1' Type B 在 v4 文档里是 **2 页 placeholder**(列字段名,不展开细则),等 hermes 走完 SOP 第一个 milestone 再 fold-back 补完。

### Q5. Charter 是 framework T0 还是 Type A T1

hermes 的 mission-charter 包含 Type A 专有字段(`acceptance.{mode,calibration}`)+ universal 字段(mission.goal,autonomy.approved_scope)。**建议**:Charter schema **base 是 T0**(任何 type 都可用),profile-specific section 是 T1(Type A 加 acceptance section,Type B 加 sop_definition section)。

### Q6. Application Guide 单一 monolithic doc 还是多 sub-docs

§5.5 我列了 12+ 子 doc。备选:做成 1 个 80-100 页 monolithic guide。**建议**:多 sub-doc,因为每个 Phase 用户读的时机不同,monolithic 难 cherry-pick;但加一份 `application-guide/00-overview.md` 做 cross-link 导航。
