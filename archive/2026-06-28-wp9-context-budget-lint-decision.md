# WP-9 — context-budget lint: measure-first baseline + design contract

## 0. Objective and non-goal (the doctrine constraint)

Make per-role / per-task **cold-start context budget** a *checkable contract* that catches
future bloat regressions — **without** violating the framework doctrine
**"sufficient context > artificially small context"**
(`process/context-passing-efficiency.md` §1.1/§1.2; the binding line is §1.2:
*"Resolve in favor of sufficient when in doubt; efficient is an optimization on top of
sufficient. A prompt that's slightly over-budget but sufficient is better than a tight
prompt that requires the receiving role to ask…"*, line 42).

WP-9 saves **0 direct tokens**. It is a **regression guardrail**. Success =
(a) it catches a future bloat regression, and (b) it **never false-hard-stops** a normal
role that legitimately needs a lot of context.

The lint is therefore **advisory**: it WARNS over a threshold, attributes the oversized
section, and supports a **waiver-with-rationale**. It HARD-STOPS only for a small,
explicit set of **structural anomalies** (an unbounded / structurally-broken load set) —
NEVER for a normal role that simply needs a lot of legitimate context.

## 2. Threshold model (deterministic + configurable)

**Primary mechanism = checked-in regression baseline snapshot + per-entry drift warning.**

- **Baseline snapshot** (`engine-kit/validators/data/context_budget_baseline.yaml`,
  generated by `context_budget_report.py --emit-baseline`): for each budget key, the
  measured `total_bytes` + `by_purpose` + per-file `{path: bytes}`. The baseline is the
  *exact current measurement at HEAD*, so at HEAD current == baseline ⇒ drift 0% ⇒
  **no entry warns** (the "no false positive at HEAD" requirement is satisfied
  structurally, not by tuning).
- **Drift warning threshold** = `DRIFT_WARN_FRACTION` (default **0.10** = +10%,
  configurable globally in the baseline file and overridable per entry). An entry WARNS
  when `current_bytes > baseline_bytes × (1 + fraction)`.

Why 10% is "real bloat, not normal variation": +10% on the smallest tracked entry
(`review`, 82,414 B) is **+8,241 B ≈ +2,060 tokens** of NEW content; on the
`governance-floor` it is **+5,953 B ≈ +1,488 tokens**. Those are doc-sized additions, not
the few-hundred-byte noise of a prose edit. A sentence added to a kernel (~100–300 B)
stays far under threshold; re-adding a whole briefing doc trips it. The threshold is a
deliberate, documented headroom, not a tight ceiling.

We deliberately do **not** use an absolute *warning* threshold (it would need per-role
tuning and risk false positives on legitimately-heavy roles like `deliver`). The absolute
figure is reserved for the **anomaly** hard-stop (§4), which is structural, not a budget.

## 3. Oversized-section attribution

When an entry is over threshold (or hits the absolute-ceiling anomaly), the report names
the responsible file(s)/section(s) by reusing `load_sizer`'s per-file + `by_purpose`
sizing:
- **per-purpose delta** — current `by_purpose` vs baseline `by_purpose` (which of
  governance / role_card / briefing grew).
- **per-file delta** — `[{path, current_bytes, baseline_bytes, delta}]` for files whose
  bytes changed or that are NEW, sorted by `(-delta, path)` (deterministic). New files
  show `baseline_bytes: 0`.

`ok` rows carry no attribution (nothing to attribute). At HEAD every row is `ok`, so the
committed trace shows no attribution; the gate test exercises attribution via a synthetic
oversized load set.

## 4. Anomaly definition (the ONLY hard-stops) — narrow, explicit, non-vacuous

An anomaly is a *structurally-broken or unbounded* load set, never a normal role that is
merely large. Anomalies are **not waivable** (see §5). The four anomalies:

| id | anomaly | condition | proven non-vacuous AND can't fire on a normal role |
|---|---|---|---|
| A1 | absolute ceiling exceeded | `current_bytes > ANOMALY_ABS_CEILING_BYTES` (default **400,000 B ≈ 100,000 tok**) | heaviest real role `deliver`=168,017 B is **2.38× below** the ceiling; a test builds a >ceiling set → fires |
| A2 | missing / zero baseline | a tracked key's baseline is absent or `≤ 0` | every real key has a positive baseline; deleting an entry → fires |
| A3 | disabled lesson bound | `lesson_selection.DEFAULT_BUDGET` has **both** L1 ceilings `≤ 0` (the WP-6 runtime bound fully disabled ⇒ the design's "unbounded lesson concatenation") | real `DEFAULT_BUDGET`=(8, 4096) is active; patching both to 0 → fires |
| A4 | missing / unreadable cold-start root | `size_role(...).missing` non-empty (a mandatory governance/role-card doc deleted/renamed without updating the load graph) | all roots resolve at HEAD (`missing=[]` for every key); deleting a root → fires |

**A3 is the static backstop for the WP-6 work.** The runtime now bounds the only
previously-unbounded injected channel (`driver._lessons_block` via
`lesson_selection.select_for_injection`). The lint statically asserts that bound is still
*structurally present*: if a future edit sets both `LessonBudget` L1 ceilings to 0
(re-enabling unbounded concatenation), A3 hard-stops. A3 is intentionally *narrow* — only
**both** ceilings disabled counts (if either the count OR the byte ceiling is positive, L1
is still bounded, so it is NOT an anomaly). The lint does not re-implement the runtime
bound; it checks the bound's structural presence.

File-level fail-safe (§7): a **missing/unparseable baseline FILE** → a single global
`baseline_unreadable` anomaly (`ok=False`); fail-closed, because the regression anchor is
the whole contract.

## 5. Waiver model (auditable, never silent, cannot hide an anomaly)

`engine-kit/validators/data/context_budget_waivers.yaml`: a deterministic, checked-in,
auditable map keyed by `(key)` (one waiver per budget key) carrying a required
`rationale`. When an entry's status would be `warn` (drift over threshold) AND a waiver
exists for its key, the status becomes **`waived`** and the report records
`waived: <rationale>` — **never silent**: a waived item still appears in the report with
its rationale, current bytes, and attribution.

Hard safety properties (tested):
- **A waiver cannot hide an anomaly.** Status resolution computes anomalies FIRST; a
  waiver is consulted ONLY to downgrade a `warn`. An entry that is both over the absolute
  ceiling (A1) *and* has a waiver stays `anomaly`. A waiver on the lesson-bound key does
  not suppress A3.
- **A waiver cannot silently hide bloat.** A `waived` row is fully reported (bytes,
  attribution, rationale). The waiver records *why* the bloat is accepted; it does not
  erase it from the report.
- At HEAD the waiver file is **empty** (no entry needs a waiver — every row is `ok`). The
  gate test exercises the waiver path with a synthetic drift + waiver.

## 6. Doctrine guard — why this cannot force-shrink sufficient context

1. **Drift is advisory + waivable.** Over-threshold = a `warn`, not a block. Two
   non-shrinking escapes clear it: (a) **bump the baseline** (a reviewed, auditable
   snapshot of the new legitimate floor) or (b) **file a waiver-with-rationale**. Neither
   asks anyone to remove context. The gate failing on an un-waived drift forces a
   *decision* (justify-and-rebaseline OR waive), it never forces a *shrink*.
2. **Hard-stops are structural only.** The four anomalies (§4) fire on broken/unbounded
   load sets, each proven (§4 table + tests) unable to fire on a normal role, with
   documented headroom (A1 is 2.38× the heaviest role).
3. **The lint never recommends removing context.** It reports size, drift, and
   attribution. The remediation it implies for legitimate growth is *record it*
   (rebaseline/waive), consistent with the doctrine's "resolve in favor of sufficient".

Therefore the lint is a **checkable accountability contract** (you must record *why* a
cold-start grew), not a **ceiling** (it never caps how much sufficient context a role may
carry).

## 7. Fail-safe behavior (deterministic, documented)

- **Baseline file missing/unparseable** → one global `baseline_unreadable` anomaly,
  `ok=False`. Fail-closed: the contract is meaningless without its anchor.
- **Baseline entry missing / `≤ 0`** for a tracked key → A2 anomaly. Fail-closed.
- **Waiver file missing** → no waivers (valid; waivers are optional).
- **Waiver file unparseable / malformed** → waivers treated as empty (`{}`) +
  a `structural_issue` recorded; warnings are **surfaced, not suppressed**. Fail-safe
  toward *surfacing* bloat — a broken waiver file can never silently hide a regression.
- All paths are deterministic: same inputs ⇒ byte-identical report (no clock, no
  randomness; JSON sorted; rows in fixed `BUDGET_ENTRIES` order; attribution sorted).

## 8. Delivery form (repo-local, no external CI assumed)

- **CLI** `engine-kit/validators/context_budget_report.py` — pure/deterministic, reuses
  `load_sizer` (no duplicate sizer logic) + reads `lesson_selection.DEFAULT_BUDGET`. Modes:
  default text report; `--json`; `--emit-baseline` (regenerate the snapshot);
  `--repo-root`, `--adopter-root` (one-off adopter-static sizing); `--strict`.
  - Exit codes: **default** exits nonzero iff an **anomaly** is present (advisory: drift
    warnings do NOT hard-stop the default CLI). **`--strict`** exits nonzero iff
    `not ok` (anomaly OR un-waived warning) — the build-gate semantics.
- **Data** (checked-in, deterministic YAML, NOT a `_sources.yaml` source and NOT a
  kernel-coverage inventory): `data/context_budget_baseline.yaml`,
  `data/context_budget_waivers.yaml`.
- **pytest gate** `engine-kit/validators/tests/test_context_budget_report.py` —
  mirrors how `kernel_equivalence.py` is both a CLI and a tested build-time gate. Asserts:
  no current key warns un-waived AND no anomaly at HEAD (`check()["ok"]`), anomalies
  hard-stop and are non-vacuous, waivers suppress + record, a waiver cannot hide an
  anomaly, attribution is correct, determinism (byte-identical), fail-safe on
  malformed/missing baseline+waiver.

## 9. Reuse + isolation invariants (what WP-9 does NOT touch)

- Reuses `load_sizer.size_role / size_load_set / role_cold_start_roots / GOVERNANCE_TRIO /
  TASK_SCOPED_COLD_START` — **no duplicate sizer logic**.
- Reads `lesson_selection.DEFAULT_BUDGET` only (the A3 structural backstop). Does not
  change WP-6 lessons tiering.
- **Additive only**: one validator + two data files + one test. No runtime / dispatch /
  governance / kernel / role-card / resolver change.
- Confirmed it does **not** touch `_acceptance_resolver_graph`, `acceptance_input_hash`,
  `_sources.yaml`, or any kernel-coverage inventory. The baseline/waiver YAML are private
  validator data, not governance sources.
