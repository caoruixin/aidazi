# charter_validator

Deterministic, no-LLM, no-network hard-kernel validator for a Δ-18 mission charter. It runs two layers: (A) **structural** validation of the charter YAML against `schemas/mission-charter.schema.json` via `jsonschema`, and (B) **semantic** rules the schema cannot express — the 9 default MANDATORY_CHECKPOINTS may not be bypassed in any of the four shapes (omitted / emptied / disabled / overridden, including auto-confirm-style semantic weakening), `tooling.acceptance.on_fix_required.human_confirm_required` must be `true`, `tooling.acceptance.on_fix_required.route_options` must be a non-empty list, `adaptive_insert` requires a `max_inserted_subsprints` bound when enabled, it WARNs when `tooling.acceptance.skills` is present while `judge_calibration.status: calibrated` (calibration corollary), and — when a campaign plan is supplied via `--campaign-plan` — a present `gap_followup` block must name both bounds (`max_subsprints` + `max_no_progress_rounds`) with `max_no_progress_rounds` PINNED to `1` (Δ-19 / §1.7-F §A.3 — an unbounded/missing bound or a non-shrink-tolerating value > 1 is a §1.7-D evasion; the static sibling of the runtime `campaign.py:_gap_followup_bounds`). CLI: `python charter_validator.py <charter.yaml> [--campaign-plan <plan.json>]` (exit 0 on pass, non-zero on any error; warnings do not fail). This is an engine-kit *implementation* only — the **normative source of these rules stays in `process/delivery-loop.md` §4.2.2 + governance/constitution.md §1.7-D** (and §1.7-C, §1.7-F, §3.6); if this file and the spec disagree, the spec wins. P-0a facets (connectors default-deny, harness×model capability gate, skill integrity) are left as no-op extension-point functions pending their schemas.

# stanza_validator

Deterministic, no-LLM, no-network validator for a sprint-stanza — the compact 4-field machine-validated header (`sprint_id`, `scope_in`, `layers`, `exit_criteria`, plus optional `modules` / `milestone_id` / `next_subsprint`) carried in the front-matter of `docs/sprint_objective.md`. It mechanizes friction case F4 (`docs/friction-playbook.md`): catch a stanza with missing or wrong-typed fields BEFORE the compact dev prompt is dispatched, not at the expensive close gate. The single structural layer validates the stanza (YAML or JSON; a JSON file parses through the same `yaml.safe_load` path) against `schemas/sprint_stanza.schema.json` via `jsonschema`, reporting each finding with a clear message and the offending path; it accepts either a bare stanza mapping or a document carrying a `sprint_stanza:` key (unwrapped automatically). CLI: `python stanza_validator.py <stanza.(yaml|json)>` (exit 0 on a schema-valid stanza, non-zero on a schema-load failure, parse failure, or any structural error). This is an engine-kit *implementation* of the orchestrator's `validate_stanza` preflight gate (`process/delivery-loop.md` §4.2.4); the **normative source stays in `schemas/sprint_stanza.schema.json`** (and the F4 discipline in `docs/friction-playbook.md`). If this file and the schema disagree, the schema wins.

Run the same way as above, substituting `stanza_validator.py tests/fixtures/valid-stanza.yaml`; both suites are discovered together by `python -m unittest discover -s tests -v`.

# adopter_wiring_validator

Deterministic, no-LLM, no-network validator for an adopter repo's **harness root-file wiring** — the check that a cold-start session actually loads the root `AGENTS.md` Control Plane entry. Claude Code auto-loads `CLAUDE.md` (not a bare `AGENTS.md`), so a Claude-Code adopter whose root holds only `AGENTS.md` starts every session without the framework's natural-language command surface. For a **Claude Code** target it requires a root `CLAUDE.md` carrying a valid line-level `@AGENTS.md` import of the **same-root** `AGENTS.md` (rejecting a missing `CLAUDE.md`/import; an absolute path, a `..` escape, or a subdirectory target — flagged even when a valid `@AGENTS.md` is also present, since an escaping import still loads an out-of-root file; a symlink redirect; an import inert inside a code block; or a `CLAUDE.md` that re-copies the governance chain instead of importing only `@AGENTS.md`); for a **Codex** target it requires only a non-empty root `AGENTS.md` (no `CLAUDE.md`); a **Cursor** target requires a real `.cursor/rules` entry — a non-empty single-file `.cursor/rules` OR a `.cursor/rules/` directory with ≥1 non-empty `*.mdc` rule (no symlink escaping root) — and **FAILs (blocking: `cursor_missing_rules` / `cursor_rules_invalid`)** when absent/empty/invalid, since a bare `AGENTS.md` is never Cursor wiring; an undeclared/`headless` harness WARNs (exit 0). The target harness is resolved by priority `explicit --harness > a single contradiction-free formal declaration > unspecified`; persistent sources are the charter's per-role `tooling.<role>.harness`/`agent_kind` (root-file harnesses only; `headless`/API roles impose no root file) and an OPTIONAL schema-free pin marker in `docs/current/adoption-state.md` (`<!-- adopter-root-harness: claude_code -->`). Two persistent sources that **contradict** — declare disjoint, non-empty harness sets sharing no common harness — FAIL (the tool surfaces the conflict and never silently picks one); partial overlap is not a contradiction (the union is validated), and an explicit `--harness` selects the invocation target without masking a genuine contradiction. CLI: `python adopter_wiring_validator.py <adopter-root> [--harness claude_code|codex|cursor|headless] [--charter PATH] [--adoption-state PATH]` (exit 0 on PASS or WARN, non-zero on any error). The **normative source of these rules stays in `governance/context_briefing.md` §1.1**; if this file and §1.1 disagree, §1.1 wins. Onboarding (`ONBOARDING.md` Step 8) runs it as part of the GREEN gate.

Run the same way as the others, substituting `adopter_wiring_validator.py ../../examples/minimal-greenfield --harness claude_code`; its suite is discovered together with the rest by `python -m unittest discover -s tests -v`.

# control_plane_validator

Deterministic, no-LLM, no-network validator for the default Control Plane Session contract. It checks that root `AGENTS.md` contains a fenced `control-plane-load` YAML block, that the default load graph names the required tiny refs under `.orchestrator/control/`, that role cards / action banks / handoffs / archives / audit transcripts / eval artifacts / broad globs are not default-loaded, and that optional `.orchestrator/control/state.json` plus `.orchestrator/control/intents.jsonl` records validate against `schemas/control-plane-state.schema.json` and `schemas/control-plane-intent.schema.json`. CLI: `python control_plane_validator.py <adopter-root> [--state .orchestrator/control/state.json] [--intents .orchestrator/control/intents.jsonl]`. This validator enforces context discipline only; routing intelligence remains LLM-assisted but schema-validated and fail-closed per `process/control-plane-routing.md`.
