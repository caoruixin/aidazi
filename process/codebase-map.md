# aidazi Codebase Map

READ-ONLY navigation index for maintainer sessions. Purpose: a new session (or a thin,
read-only session-start primer) reads this to INHERIT structural understanding of the codebase
instead of re-scanning the whole tree — saving tokens and repeated re-analysis. It is a reading
aid ONLY: it never gates, blocks, or authorizes anything, and nothing here is required in order
to edit, commit, run, spawn subagents, use worktrees, or work across repos. If the map is absent,
stale, or unhelpful, ignore it and search the code normally.

How to use it:
- The fenced ```json block below is the machine-readable index: one entry per area, with `covers`
  (paths the area spans), `anchors` (path:symbol jump targets), `tests`, `canonical_docs`,
  `keywords`, `depends_on`, and a one-line `responsibility`.
- To localize a task: match the task against `keywords`/`responsibility`, then open the selected
  areas' `anchors`/`canonical_docs` directly instead of grepping the whole repo.
- `map_checkpoint` is the commit this map was last verified against. To find what may be stale,
  run `git diff <map_checkpoint>..HEAD --name-only` and re-read only the touched areas. On a large
  diff, an unmappable task, or any doubt, fall back to normal code search — never block on the map.
- The prose bodies after the JSON add data-flow / invariants for the heavier areas.

Self-contained: needs no tool, validator, or hook to be useful. `covers` globs are descriptive
navigation hints, not an enforced partition.

## Section bodies (human reference; the JSON index above is authoritative for nav targets)

### orchestrator-driver
Deterministic kernel state machine over the checkpoint inbox; verdicts validated against JSON
schemas (**fail-closed** on violation). Role routing from `charter.tooling.<role>`; fix-rounds
bounded by `charter.budget` + spawn-count guards; resume from `.orchestrator/state.json`. To change
loop control -> `driver.py:Driver`; to change a verdict contract -> also update `schemas/` + the
adapter. Invariant: the driver is the SINGLE verdict-validation point (adapters never pre-validate).

### orchestrator-e2e-acceptance
Executor *runs* evidence and emits observations; Acceptance reads verdicts from artifacts, never
the reverse. `acceptance_input_hash` / LOAD-CLOSURE is the load-bearing invariant (no unbound
verdict-affecting input). Offline `LocalHttpExecutor` is deterministic (no clock in artifacts);
Playwright path env-gated. To touch acceptance reuse -> `e2e_stage.py:check_acceptance_consistency`.

### scheduling-runloop / adapters-harness-bridge
`run_loop.py` is the cron/CI entrypoint (NOT a harness scheduler): validate charter -> `build_adapters`
(MockAdapter default; real gated by `AIDAZI_ALLOW_REAL_ADAPTER=1`) -> `Driver.run()` or `--campaign`.
Adapters expose uniform `spawn(...)->dict`; prompt via **stdin** (no argv injection); `sandbox`->
permission-mode mapping is **fail-closed**. To add a harness -> `adapters/__init__.py:resolve_adapter_class` + a new adapter subclass + registry.

### core-validators / adoption-budget-roleconfig
All deterministic (no LLM/net/clock). `charter_validator` encodes §1.7 + 9 MANDATORY_CHECKPOINTS +
gap-followup bounds. `context_budget_report.py` is the **WP-9 advisory lint** (reuses `load_sizer`;
baseline+waivers in `validators/data/`; warns on drift, hard-stops only structural anomalies).

### constraint-gate / tier0-kernel-projections / canonical-specs
`kernel_equivalence.py` proves the always-load kernels (`constitution-core`/`authoring-kernel`)
carry every canonical constraint NON-vacuously, source-hash bound to `_sources.yaml`. Acceptance
LOAD-CLOSURE proven by `acceptance_load_closure.py`. **Spec is source of truth** — on code/spec
disagreement the spec wins; changing a governance doc requires re-checking `_sources.yaml`.
